# ADR-XXXX — `CacheService` and `graphbase` stay separate stores, not one merged cache

| | |
|---|---|
| **Status** | Proposed |
| **Date** | 2026-06-30 |

## Context

Two Mongo-capable graph stores exist in this codebase and, on a first
read, look redundant:

- **`CacheService`** (`codecarto/services/cache_service.py`) — filesystem-
  first, content-addressed cache keyed by
  `sha256(url :: mode :: layout :: extensions)[:16]`, scoped per repo,
  with an index capped at 50 entries backing the UI's "Recent" panel.
  Its job is "don't reparse/refetch what's already been seen."
- **`graphbase`** (`graphbase/src/main.py`, mounted at `/db/*`) — a
  general-purpose, user-named graph store with full CRUD
  (`POST /graph/insert`, `GET /graph/{name}`, `GET /graphs`,
  `DELETE /graph/{name}`, plus node/edge-level operations). Its job is
  "save this graph under a name I chose, durably, until I delete it."

When `MONGODB_URI` is set, `CacheService` reuses graphbase's `get_db()`
connection (one `MongoClient`, same database) rather than opening a
second connection — so the two systems already share infrastructure.
That shared plumbing is what makes "just merge them" look tempting.

## Decision

Keep them as two separate abstractions sharing one Mongo connection,
rather than merging into a single keyed store:

- `CacheService` entries are derived and disposable — its key is a hash
  of parse *inputs*, not a name a person chose, and the 50-entry cap plus
  TTL (`CC_CACHE_TTL`, default 24h) assumes anything in it can be
  regenerated by re-running the same parse.
- `graphbase` entries are intentional and durable — a name the user
  picked, no TTL, no eviction beyond explicit `DELETE`.

These are different lifecycle contracts. Forcing them through one keyed
store would mean either giving cache entries TTL-defeating permanence, or
giving user-saved graphbase entries surprise eviction.

## Consequences

- `graphbase`'s `/db/*` endpoints have **zero frontend integration** as of
  this writing — the backend CRUD is complete but nothing in `web/src`
  calls it. The "recent graphs" UI today only talks to `CacheService`'s
  filesystem/Mongo-backed cache. Wiring up save/load against `/db/*` is
  tracked as its own piece of work, not a reuse of the existing
  cache-recall code path.
- Anyone adding a "save my graph" feature should reach for `/db/*`, not
  extend `CacheService` with a no-TTL mode — the moment a cache entry
  needs to survive eviction by user intent, it has become a graphbase
  entry, not a cache entry.

## Alternatives considered

1. **Merge into one Mongo collection with a `kind` discriminator field**
   (cache vs. saved) — rejected for now: the eviction/TTL logic and the
   naming/addressing logic are different enough that a discriminator field
   would mostly stand in for two services already cleanly separated by
   their HTTP surface (`/parse/cache/*` vs `/db/*`).

## Revision triggers

- Frontend integration work against `/db/*` begins and surfaces a real
  need to share lookup/list code between the two stores.
- A third Mongo-backed store is proposed, forcing a decision about whether
  this pattern (separate stores, shared connection) generalizes or breaks
  down.
